# In-box / Out-box Harness 分析框架

来自论文《Agent Harness Engineering: A Survey》第①篇阅读后的讨论。

## 定义

**In-box Harness**：基模团队在设计模型产品时，把 System prompt、上下文工程、编排等直接打包进产品内的那部分 Harness。例如 Claude Code 和 OpenAI Codex 内置的提示词体系、工具调用格式、上下文管理策略。

**Out-box Harness**：In-box 以外的一切 Harness。开发者需要自行搭建的执行沙箱、可观测性、治理与安全、工具协议集成、CI/CD 流水线等。

## 核心判断

**In-box 能力增强会催生新的 Out-box 需求。**

这不是零和博弈——In-box 把一部分问题变简单了，但同时也把「剩下没被吸收的那些问题」的重要性推高了一个级别。

## 历史演进规律

```
时间轴 →

In-box 吸收 Out-box          Out-box 催生新需求
     ↓                            ↓
工具调用格式原生支持    →    需要更细粒度的权限模型
上下文窗口扩大到 1M     →    需要更复杂的记忆检索策略
结构化输出原生支持      →    需要更强大的输出验证机制
基础编排内建           →    需要多系统集成治理
模型能执行代码         →    需要完善的沙箱和权限管控
模型能运行数小时       →    需要状态持久化和故障恢复
```

## 螺旋上升四步模型

1. 模型团队把当前 Out-box 的「瓶颈工程」吸收进模型产品（In-box 扩大）
2. 开发者因此可以构建更复杂的 Agent 系统（能力上限提升）
3. 能力上限提升暴露新的瓶颈——这些瓶颈大概率在治理、安全、可观测性这类基础设施侧（新的 Out-box 出现）
4. 这些新的 Out-box 中有一部分最终又会被模型团队吸收（回到第 1 步）

## 类比：云计算演进

```
本地部署 → 托管服务(IaaS/PaaS) → 更复杂的治理(IAM/VPC) → Serverless → 更精细的权限和可观测性
```

每一次「In-box 吸收」都会把一部分问题变简单，但同时把「剩下没被吸收的那些问题」的重要性推高一个级别。

## 在具体场景中的应用

问自己：**现在做的东西，瓶颈在哪一侧？**

- 从 0 到 60 分：In-box 的边际收益更大（改 System prompt 可能提升 20%）
- 从 60 到上线运行：Out-box 是瓶颈（治理、安全、可观测性决定能不能上线）

## 衍生思考

- In-box 决定你能做什么，Out-box 决定你做到什么程度还能收得住
- Claude Code 把 In-box 做到极致，它的 Out-box 就变得很薄
- 基于 API 自建 Agent 平台的公司，Out-box 才是主战场
- 强模型 + 弱 Harness → 危险（失控风险高）
- 弱模型 + 强 Harness → 安全但可能不够灵活
- 边界移动的方向取决于你观察的是哪个阶段、哪个产品
- 论文的三个核心证据（Bölük +13.7 百分点、Trivedy 10×、Meta-Harness 76.4%）全是 In-box 改造，但论文花了 71 页铺 Out-box 的分类体系——存在错位
