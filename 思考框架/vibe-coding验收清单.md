# 🎯 Vibe-Coding 验收清单模板

> 产品经理负责产品定义，AI 托管开发和测试。验收不再是"查代码"，而是查结果的可观测性和可靠性。
> 功能验收是及格线，日志和 trace 验收才是分水岭。

---

## 使用方式

每次验收 AI 产出的功能时，对照以下清单逐项打勾。**⛔ 表示强制项**——不通过不放过。

---

## 一、功能验收（及格线）

### ⛔ 主流程跑通
- [ ] 核心用户路径是否完整走通？
- [ ] 边界输入是否返回合理结果？（空值、超长、特殊字符）
- [ ] 错误输入是否有友好的错误提示？

### UI / 交互
- [ ] 前端是否符合产品定义中的交互要求？
- [ ] 加载态、空态、错误态是否都有处理？
- [ ] 响应式布局是否正常？

## 二、日志验收（分水岭）

### ⛔ 关键操作有日志
- [ ] 用户触发的每个核心操作有没有 log？
- [ ] 关键数据变更有没有 log？（创建 / 更新 / 删除）
- [ ] 外部 API 调用的入参和出参有没有 log？

### ⛔ 错误能定位
- [ ] 所有异常 / 报错分支是不是都有 log？
- [ ] 日志里是否包含**上下文信息**（用户 ID、请求 ID、操作对象）？
- [ ] 日志级别是否合理？（ERROR 不要当 INFO 打，WARN 不要沉默）

### 日志质量
- [ ] 日志能不能直接帮助定位问题，而不仅仅是"something went wrong"？
- [ ] 敏感信息（密码、token、PII）不会被打印到日志？

## 三、链路追踪验收（运维分水岭）

### ⛔ Trace ID 贯穿
- [ ] 从前端请求到后端处理，Trace ID 是否贯穿全链路？
- [ ] 出错时能否通过 Trace ID 从用户端一路查到数据库？

### ⛔ 跨服务/跨组件可追踪
- [ ] 多个服务之间调用是否传递了 trace context？
- [ ] 异步任务（队列、定时任务）是否有独立的 trace？
- [ ] 第三方 API 调用是否有 trace？

## 四、状态与边界验收（坑最多的地方）

### ⛔ 并发状态
- [ ] 同一用户的重复请求会不会导致状态冲突？
- [ ] 竞态条件有没有处理？（例如同时点击两次提交）

### ⛔ 异常恢复
- [ ] 服务重启后，中间状态有没有丢失？
- [ ] 部分失败场景有没有兜底？（更新 A 成功、更新 B 失败 → 事务 / 补偿逻辑）

### 边界条件
- [ ] 数据为空时的行为是否合理？
- [ ] 数据量极大时（分页、超时）表现是否正常？
- [ ] 权限边界——不同角色看到的内容 / 能做的操作是否正确？

## 五、安全验收（红线）

### ⛔ 注入防护
- [ ] 用户输入是否经过清洗 / 参数化？
- [ ] NoSQL 注入、SQL 注入有没有防护？

### ⛔ 权限校验
- [ ] 是否有越权漏洞？（A 用户能否访问 B 用户的数据？）
- [ ] 接口是否有未授权访问？

---

## 验收决策参考

| 验收结果 | 动作 |
|---|---|
| ⛔ 项全部通过 + 功能项通过 | ✅ 可以发布 |
| ⛔ 项有一项不通过 | ❌ 阻挡发布，打回 AI 修复 |
| ⛔ 项通过但功能项有缺 | ⚠️ 视缺失严重程度决策，记录已知问题 |
| 全部通过但感觉不对 | 🔍 相信直觉，再测一遍关键路径 |
