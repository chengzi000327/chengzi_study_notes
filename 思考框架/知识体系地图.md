# 🗺️ 橙子的知识体系地图

> 跨论文原理网络。每一层 = 一个底层原理，下一级 = 二级概念/具体实现。
> 新知识进来时，定位到对应原理节点下。

---

## 一、基础层：AI Agent 的核心约束

### 原理1：上下文窗口是硬约束
背后逻辑：LLM 能同时看到的信息量有限，所有设计都必须在这个约束下做优化。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 上下文即内存 | MemGPT | 像 OS 管理虚拟内存一样管理上下文，swap 换入换出 |
| 渐进式披露 | Agent Skills 综述 | 元数据预加载，触发时才加载完整正文 |
| 约束解码 | ToolGen | 只允许生成真实存在的工具 token，幻觉归零 |

**↔ 关联：** 上下文管理（C层）← 直接影响工具选择（T层）

### 原理2：状态管理决定Agent一致性
背后逻辑：Agent 能否记住并保持一致，取决于状态的存储、更新和遗忘机制。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 记忆分层 | Generative Agents | 短期→长期→抽象，三层记忆架构 |
| 遗忘机制 | MemoryBank | 艾宾浩斯遗忘曲线，时间衰减 |
| ADD/UPDATE/DELETE | Mem0 | 不是记住一切，是记得住该记得的 |
| 文件系统即状态 | Claude Code | 把状态存在文件系统里而非内存里 |

**↔ 关联：** 状态管理（C层）← 决定编排（L层）能否跨会话协作

## 二、模型层：Agent 的能力边界

### 原理3：工具模型比语言模型更关键
背后逻辑：决定 Agent 行为上限的不是模型本身，而是模型外面的包装（Harness）。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| Binding-constraint 论点 | Agent Harness Engineering | 改工具格式比换模型效果更显著（10x提升） |
| 检索即能力 | Gorilla | 离线训练追不上线上变化，推理时检索更可靠 |
| 检索生成一体化 | ToolGen | 拆两步不如合成一步，消除误差累积 |

**↔ 关联：** Harness（ETCLOVG全部七层）← 约束条件决定了模型能发挥多少能力

### 原理4：自我学习优于人工标注
背后逻辑：让模型自己学会何时调用工具、何时更新记忆，替代人工规则。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 困惑度筛选 | Toolformer | 工具调用必须"真的帮到了语言建模"才算有效 |
| 数据说话 | Toolformer | 不问用户要什么，看行为上的实际受益 |
| 记忆命中率 | Mem0 | 召回是否符合用户预期，数据驱动优化 |

**↔ 关联：** 自我学习（T层+Skills层）← 依赖评估（V层）来验证效果

## 三、技术层：多Agent系统的工程实践

### 原理5：角色分工是协作的基石
背后逻辑：多Agent不是堆更多模型，而是定义清楚谁做什么。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 角色提示 | CAMEL | "Never flip roles!" 维持角色边界的提示设计 |
| SOP驱动 | MetaGPT | 工作流规范化，每个Agents的输入/输出格式事先定义 |
| 容错循环 | AutoGen | 能发现错误并自我修复，比一次正确更重要 |

**↔ 关联：** 角色（L层）← 每个角色的工具权限由（G层）定义

### 原理6：结构化输出减少级联错误
背后逻辑：Agent之间传递标准格式的交付物，而非自由文本，防止级联幻觉。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 标准化中间产物 | MetaGPT | PRD→设计→任务→代码→测试，每步有验证点 |
| 共享消息池 | MetaGPT | 减少无效通信，Agent订阅自己关心的消息 |
| 编排优于自治 | 面试题分析 | 好PM设计的是让不聪明Agent正确协作的框架 |

**↔ 关联：** 标准化（T层+L层）← 可观测性（O层）帮助验证中间产物质量

## 四、应用层：产品视角

### 原理7：信任来自一致性
背后逻辑：用户信任AI不是因为它聪明，而是因为它稳定可靠。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 记忆一致性 | Generative Agents | 昨天答应的事今天还记得 |
| 品牌一致性 | AI助手功能边界题 | 闲聊/情感陪伴不能破坏工具产品的品牌定位 |
| 容错体验 | 多Agent面试题 | 降级路径也须输出用户可见的完整回复 |

### 原理8：安全是设计内嵌，不是事后补丁
背后逻辑：Agent Skills 26.1% 存在安全漏洞说明安全审核是平台级责任。

| 二级概念 | 论文/项目 | 关键洞察 |
|---|---|---|
| 分级授权 | Agent Skills 综述 | 按来源可信度授予权限 |
| 五层防护 | AI助手功能边界题 | 实时安全管道→危机干预→人格护栏→用户控制→风险仪表盘 |
| 三阶段灰度 | AI助手功能边界题 | 冷启动1%→扩量5-20%→全量，每阶段有退出阈值 |

**↔ 关联：** 安全（G层）← Toxicity/Bias/Prompt Injection指标（V层）是评估安全的量化标准

### 原理10：AI商业化的本质是价值定价，不是成本定价
背后逻辑：Token计费是供给侧成本逻辑（我消耗了多少计算资源），用户愿意付钱的前提是需求侧价值逻辑（这个结果值多少钱）。把推理成本封装成用户可感知、愿支付的结果价值，是AI产品商业化的核心命题。

| 二级概念 | 项目/来源 | 关键洞察 |
|---|---|---|
| 成本计价 vs 价值计价 | 我的想法笔记 | 定价锚点从「每次查询消耗多少token」转向「每个任务解决什么问题」——用户不为计算买单，为结果买单 |
| 用户心智重构 | 商业模型分析 | 和价值定价并行的卡点——用户需要从「为计算付费」转变为「为任务完成付费」，这需要产品在定价心智、交互心智、信任积累三层面累积 |
| 任务变现取代流量变现 | 商业模型分析 | AI产品从卖注意力（广告）/卖使用权（会员）转向卖效果（任务完成/问题解决）——交付逻辑变了，定价逻辑也必须变 |
| 单位经济模型闭环 | 商业模型分析 | 终极竞争力 = 在可控推理成本下持续交付高价值任务，形成正向LTV/CAC循环 |

**↔ 关联：** 商业模型（G层）← 价值定价依赖评测（V层）量化交付质量，形成信任基础；用户信任（原理7）是价值定价的前提条件——用户只有相信AI稳定可靠，才愿为结果付费

## 五、验证层：连接理论与实践的桥梁

### 原理9：评测可信度由评估链质量决定
背后逻辑：LLM应用的评测不是打分题，是多层验证系统——测试用例的覆盖度（产品判断）、judge模型的可靠性（工程选择）、样本量带来的统计效力（统计学基础），每个环节的短板都会让最终的指标数字失去意义。

| 二级概念 | 项目/来源 | 关键洞察 |
|---|---|---|
| 测试用例是底座 | DeepEval实践分析 | 工具解决"怎么跑"和"怎么看"，不解决"测什么"——20个高质量用户场景用例 > 200个随机构造用例 |
| LLM-as-Judge偏差 | DeepEval批判 | 用模型A评判模型B的输出，存在循环论证——judge自身的幻觉、偏见、prompt顺序敏感性未被充分正视 |
| 聚合指标掩盖子指标 | RAGAS拆解 | 当RAGAS从0.78涨到0.82，你不知道是检索（Precision/Recall）变好了还是生成（Faithfulness）变好了——必须拆解看子指标驱动优化 |
| 小样本陷阱 | 统计效力基础 | 50条测试用例下的指标均值几乎没有统计效力——需要置信区间和误差棒，数据分布比均值更有信息量 |
| 人机混合验证 | 生产环境实践 | 100%自动化评判不可靠，关键场景需人工校验 + Multi-judge投票降噪 + 一致性校准 |

**适用指标分组：**
- **RAG场景：** Faithfulness（核心） + Context Precision/Recall（检索侧诊断） + Answer Relevancy
- **Agent场景：** Tool Correctness + Task Completion + 多轮Coherence/Consistency
- **安全场景：** Toxicity + Bias + Prompt Injection
- **通用质量：** G-Eval（多维度文本质量打分，无context依赖）

**↔ 关联：** 评测（V层）← 连接所有层。"测什么"由产品层（G层）定义目标用户场景；"怎么测"由技术层（T/L层）选择工具和judge方法；"测出来的数据"反馈回基础层（C层/状态管理/工具选择）驱动改进闭环

## 跨层核心循环

```
基础层约束（上下文/状态）→ 模型层边界（工具/自学习）
→ 技术层实践（角色/结构化）→ 应用层产品（信任/安全）
→ 反馈回到基础层（可观测性数据驱动改进）
```

**这个循环每走一圈，你的知识体系就更像一个专家的原理网络。**
